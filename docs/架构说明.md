# 架构说明（Architecture）

> 版本：0.1  
> 状态：与当前代码结构一致（M1 已实现）

本文描述 **系统如何拆层、数据如何流动、边界在哪里**。产品需求与字段级数据模型仍以 `开发规划方案.md` 为 SSOT。

---

## 1. 目标与非目标

### 1.1 目标

- **规则优先**：可吃/不可吃/何时保守，由规则引擎 + 配置数据裁决；LLM 只负责解释与编排（在宿主侧）。
- **可记忆**：档案、计划、反应、食材状态、审计可追溯（SQLite）。
- **可测试**：规则、状态机、关键工具链路可单测/集成测（Vitest）。
- **可演进**：M2/M3 能力主要体现为新增工具与仓储方法，而不是推翻核心模型。

### 1.2 非目标（当前阶段）

- 不在核心运行时引入 Python 决策链（与 SSOT 一致：Python 仅可选脚本）。
- 不实现云端多租户同步（本地 SQLite 优先）。
- 不把「医疗诊断/急救决策」产品化（见 SSOT 边界）。

## 2. 逻辑架构（分层）

采用 SSOT 中的四层思想，并在仓库里落到具体目录：

```text
┌───────────────────────────────────────────────┐
│  Skill / Agent 层（宿主侧）                      │
│  SKILL.md + prompts/*.md                       │
│  负责：教模型何时调用工具、输出模板、风险提示话术   │
├───────────────────────────────────────────────┤
│  工具层 `src/tools/`*                           │
│  负责：把“用户意图”落成事务：读写 DB、组装响应     │
├───────────────────────────────────────────────┤
│  规则引擎层 `src/rules/`*                        │
│  负责：月龄频次/质地/过敏阻断/计划编排/状态机      │
├───────────────────────────────────────────────┤
│  记忆层 `src/storage/*` + `data/*.json`         │
│  负责：SQLite schema、migrations、repositories   │
│        JSON：食材库、阶段表、规则参数、菜谱模板    │
└───────────────────────────────────────────────┘
```

**依赖方向（必须单向）**

- `tools` → 允许依赖：`storage`、`rules`、`config`、`types`
- `rules` → 允许依赖：`config`、`types`、纯函数工具（如 `dates.ts`）
- `rules` → **禁止**依赖：`tools`（避免循环）、默认也**不**依赖具体 DB 连接（需要历史数据时由 `tools` 查询后注入）
- `storage` → 允许依赖：`types`（仅类型）与 SQL/迁移；禁止依赖 `rules`

## 3. 物理结构（仓库映射）

```text
src/
  index.ts              # 对外 API（库入口）
  cli.ts                # 本地调试 CLI（非宿主唯一入口）
  runtime.ts            # db + 配置数据 bundle 组装
  version.ts            # 规则/实现版本号（写入 meal_plan.source_version）
  dates.ts              # 日期/月龄计算（纯函数）
  config/               # 读取与校验 data/*.json（zod）
  rules/                # 规则引擎与状态机
  storage/              # DB 连接、migrations、repositories
  tools/                # “工具函数”实现（未来可桥接 MCP/OpenClaw tool）
  prompts/              # 给模型的提示词片段（不参与裁决）
data/                   # 配置化规则与食材/模板（版本控制）
tests/                  # vitest：规则单测 + 集成测
docs/                   # 文档 SSOT + 工程文档
SKILL.md                # AgentSkills/OpenClaw 技能入口
manifest.json           # 机器可读元数据（辅助集成）
```

## 4. 关键数据流（M1）

### 4.1 建档（`createOrUpdateBabyProfile`）

1. `tools/profile` 校验/生成 `baby_id`（若未提供）。
2. `storage/repositories/baby-profile` 写入 `baby_profile`。
3. `active` 策略：激活宝宝时先清空其他 `active`（M1 单宝宝主路径；M2 将扩展为多宝宝切换策略）。

### 4.2 周计划（`generateWeeklyMealPlan`）

1. `tools/meal-plan` 读取档案 + `food_status` + 近期 `food_trial`。
2. `rules/engine.runPrePlanChecks` 产出审计条目与 `block_new_foods` 等约束。
3. `rules/meal-planner.buildWeeklyMealPlan` 生成 7 天菜单候选。
4. 事务写入 `meal_plan` + `meal_item`，并写入 `rule_audit_log`。

### 4.3 反应记录（`logFoodReaction`）

1. 追加 `food_trial`。
2. `rules/state-machine` 更新 `food_status`。
3. 若严重/高风险关键词命中：`blocker` 逻辑会影响后续计划生成（见集成测试）。

### 4.4 ICS 导出（`exportCalendarIcs`）

1. 读取 `meal_plan`/`meal_item`。
2. 生成 RFC5545 文本（当前为偏实用的固定 UTC 餐点时间策略；时区参数为后续增强点）。

## 5. 扩展点（为 M2/M3 预留）


| 扩展点       | 建议落点                                 | 说明                                    |
| --------- | ------------------------------------ | ------------------------------------- |
| 新工具       | `src/tools/*.ts` + `src/index.ts` 导出 | 保持“薄工具 + 厚规则/厚存储”                     |
| 新规则组      | `src/rules/*` + `data/*.json`        | 先加配置与单测，再接入 engine                    |
| schema 变更 | `src/storage/migrations/*`           | 追加迁移文件；禁止静默改历史表语义                     |
| 宿主集成      | 外层 adapter                           | 本仓库提供库/CLI；由 OpenClaw/Gateway 映射 tool |


---

## 6. 风险与约束（架构视角）

- **规则与模型输出冲突**：以规则引擎为准，并要求审计可解释（`rule_audit_log`）。
- **多宝宝隔离（M2）**：所有写路径必须显式 `baby_id`；禁止仅靠昵称隐式写入。
- **native 模块**：`better-sqlite3` 依赖预编译/本机构建；CI 需覆盖目标平台。
