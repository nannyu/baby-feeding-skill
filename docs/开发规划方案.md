## 2. 项目概述

### 2.1 项目名称

宝宝辅食（Baby Feeding Planner for OpenClaw）

### 2.2 项目定位

面向多孩家庭的婴幼儿辅食规划、排敏管理、食物反应记忆与苹果生态联动技能。

本项目不是静态菜谱生成器，而是一个**科学辅食操作系统**。它的核心差异在于：

- **规则驱动**：真正决定可不可吃、何时吃、如何吃的是规则引擎，不是 LLM 即兴生成。
- **可记忆**：系统会持续记住“这个宝宝吃过什么、反应如何、爱不爱吃、是否需要暂停”。
- **多宝宝隔离**：每个宝宝有独立档案、独立食材状态、独立计划历史与独立风险记录。
- **可联动**：支持把计划进一步转成采购、日历、提醒事项和备忘录工作流。

### 2.3 背景与机会

- 辅食规划是高频、长期、强记忆依赖任务，涉及月龄、体重、出牙、过敏风险、质地进阶、家庭烹饪习惯等多个变量。
- 通用菜谱产品通常只提供静态菜单，难以持续管理“个体化耐受史”和“动态调整”。
- OpenClaw 具备技能、工具与自动化能力，适合将建档、计划、排敏、记录、采购和苹果生态联动整合为一个持续运行的 agent。

## 4. 适用范围、边界与安全红线

### 4.1 适用范围

- 6-24 月龄婴幼儿的辅食安排
- 新食材引入与排敏管理
- 食物反应记录与后续调整
- 周计划与采购清单生成
- 苹果日历、提醒事项、备忘录类联动输出

### 4.2 不适用范围

- 疾病诊断
- 急性过敏急救替代决策
- 处方、药品、医疗建议
- 特殊疾病营养治疗方案替代

### 4.3 安全红线

一旦记录到以下高风险表现，系统必须停止继续推荐新食材，并进入风险提示流程：

- 呼吸困难
- 口唇或舌部肿胀
- 持续剧烈呕吐
- 明显精神差、反应差

### 4.4 风险提示原则

- 系统只能给出“停止尝试、立即关注、尽快就医/咨询医生”的风险提醒，不能伪装成医疗诊断。
- 对高风险情形，输出优先级必须高于菜谱推荐内容。
- 若用户主动要求继续尝试已被阻断的高风险食材，系统应显式提示该行为存在风险，不应默认放行。

## 5. 规则依据与优先级

### 5.1 规则基线


| 优先级 | 来源                    | 适用范围                        | 系统作用      |
| --- | --------------------- | --------------------------- | --------- |
| P0  | WHO 辅食添加指南            | 6-23 月龄辅食添加、频次、多样化、安全与响应式喂养 | 全局规则基线    |
| P0  | AAP / HealthyChildren | 高敏食物引入、过敏预防、观察与风险提示         | 排敏与过敏规则   |
| P0  | 中国营养学会婴幼儿喂养指南及相关官方指南  | 中文语境下的年龄分层、食物类别、本地化建议       | 中国家庭本地化规则 |
| P1  | 《西尔斯育儿百科》             | 家长易理解的经验表达与解释               | 辅助解释层     |
| P2  | 宝宝个体历史数据              | 已尝试食材、反应、爱吃度、便便与质地接受度       | 个体化动态调整   |


### 5.2 冲突处理原则

- 若书籍经验与 WHO/AAP/中国营养学会指南冲突，以官方/学术组织指南为准。
- 若通用指南与宝宝个体已知严重不耐受或疑似过敏记录冲突，以个体风险记录为准。
- 若家长或医生明确要求禁用某食材，系统应写入 `MANUAL_BLOCK`，除非用户主动解除。

- 同时管理两个及以上宝宝饮食的多孩家庭

### 7.3 多宝宝能力要求

- 每个宝宝必须拥有独立 `baby_id`、档案、耐受食材列表、禁用食材列表、反应日志和计划历史。
- 若用户未明确指定对象，系统默认作用于最近使用的宝宝，但输出中必须显式标注当前对象。
- 所有写操作必须绑定明确的 `baby_id`，不得仅依赖自然语言上下文猜测。
- 家庭层共享词典（如常备食材、烹饪偏好）可以复用，但不得覆盖宝宝个体规则。

## 9. 总体技术方案

### 9.1 架构总览

建议采用“OpenClaw Skill + Nutrition Rules Engine + Memory Store + Apple Bridge”四层架构：

```text
┌─────────────────────────────────────────────────┐
│               Skill 层 (SKILL.md + tools)       │
│       对话入口、上下文组装、命令解释、输出编排     │
├─────────────────────────────────────────────────┤
│          规则引擎层 (TypeScript service)         │
│     月龄规则、排敏逻辑、菜谱编排、风险阻断裁决     │
├─────────────────────────────────────────────────┤
│           记忆层 (SQLite + JSON config)          │
│       宝宝档案、进食日志、状态机、计划历史         │
├─────────────────────────────────────────────────┤
│         桥接层 (ICS / Shortcuts / local script)  │
│      Calendar / Reminders / Notes 渐进增强联动    │
└─────────────────────────────────────────────────┘
```

### 9.2 核心技术决策


| 维度     | 方案               | 说明                               |
| ------ | ---------------- | -------------------------------- |
| 主语言    | TypeScript       | 与 OpenClaw 生态一致，类型安全，适合规则和工具接口实现 |
| 运行时    | Node.js          | 作为核心运行时                          |
| Python | 可选辅助脚本           | 仅用于本地数据整理、导出辅助或离线处理，不承担核心规则决策    |
| 包管理    | pnpm             | 安装快、磁盘占用低                        |
| 构建     | tsup             | 简洁稳定，适合技能打包                      |
| 存储     | SQLite + JSON 配置 | 本地优先、零部署、便于备份与迁移                 |
| 测试     | Vitest           | 适合 TypeScript 单测与集成测试            |
| 数据策略   | 配置化规则表 + 强校验     | 减少硬编码，便于持续扩充                     |

### 9.4 设计原则

- 核心规则必须由规则引擎裁决，LLM 只负责表达、解释和交互。
- 外部联动能力是增强项，不得影响基础计划生成。
- 规则、食材、模板优先配置化，而不是把业务逻辑散落在 Prompt 中。
- 数据模型需优先服务“多宝宝隔离”“审计可追溯”“动态调整”三件事。

### 10.3 食材状态机

```text
UNTRIED
  -> PLANNED
  -> OBSERVING
  -> TOLERATED
  -> DISLIKED
  -> SUSPECTED_INTOLERANCE
  -> SUSPECTED_ALLERGY_BLOCK
  -> MANUAL_BLOCK
```


| 状态                      | 含义          | 对计划的影响      |
| ----------------------- | ----------- | ----------- |
| UNTRIED                 | 未引入         | 可被排敏计划选中    |
| PLANNED                 | 已进入计划，尚未食用  | 占用排敏位，不重复安排 |
| OBSERVING               | 已食用，处于观察窗口内 | 不叠加新高敏食材    |
| TOLERATED               | 已耐受         | 可进入常规菜单     |
| DISLIKED                | 无过敏但明显不爱吃   | 降低频率或换做法    |
| SUSPECTED_INTOLERANCE   | 疑似不耐受       | 暂停并安排复试     |
| SUSPECTED_ALLERGY_BLOCK | 疑似过敏        | 停止自动安排并标红   |
| MANUAL_BLOCK            | 人工禁用        | 不可安排，除非用户解除 |

## 11. 规则引擎设计

### 11.1 核心原则

**规则引擎优先于 LLM。**  
LLM 负责自然语言交互、菜谱表达、说明与解释；真正的约束条件由配置化规则表和 TypeScript 规则引擎决定。

### 11.2 输入要素

规则引擎的输入至少应包括：

- 月龄
- 体重
- 出牙状态
- 喂养方式
- 已耐受食材
- 禁用食材
- 近期进食反应
- 家庭烹饪偏好
- 当前可用食材

### 11.3 输出前的三道校验

1. 年龄与质地校验：食材形态是否符合当前月龄与接受度
2. 风险与禁用校验：是否触碰高风险状态、禁用食材或年龄禁忌
3. 重复与多样性校验：近期食材覆盖是否过窄，是否出现不合理重复

### 11.4 规则组


| 规则组     | 输入             | 输出                 | 配置来源                  |
| ------- | -------------- | ------------------ | --------------------- |
| 年龄/频次规则 | 月龄、喂养方式        | 每日辅食次数范围           | `feeding-stages.json` |
| 质地规则    | 月龄、出牙、接受度      | 泥、碎、软块、finger food | `texture-rules.json`  |
| 排敏规则    | 高敏食材、风险等级、历史状态 | 新食材顺序与观察窗口         | `allergen-rules.json` |
| 多样化规则   | 近 7/14 天食材覆盖   | 轮换建议               | 引擎逻辑                  |
| 偏好调整规则  | 爱吃度、拒食次数       | 保留、降频、换做法          | 引擎逻辑                  |
| 阻断规则    | 严重反应、禁用状态、年龄禁忌 | 直接排除并提示            | `allergen-rules.json` |


### 11.5 风险阻断策略

若命中高风险症状，系统必须：

1. 停止继续推荐新食材
2. 将相关食材状态更新为 `SUSPECTED_ALLERGY_BLOCK` 或更严格状态
3. 输出高优先级风险提示
4. 记录审计日志
5. 不自动清除阻断，除非用户明确解除或补充说明

## 13. 指令设计与输出规范

### 13.1 系统级提示词原则

- 始终先识别当前作用对象；若上下文不明确，优先引用 `active_baby`。
- 所有食谱结果都应包含“规则依据摘要、风险提示、观察点、可替代方案”。
- 对新食材或高敏食材，必须安排白天观察窗口，不与多个新食材叠加。
- 若存在疑似过敏或高风险记录，应降低自由生成程度，优先调用排敏与风险工具。

### 13.2 用户指令分类


| 类别   | 示例指令                      | 预期动作      | 优先工具                            |
| ---- | ------------------------- | --------- | ------------------------------- |
| 建档   | 给宝宝“朵朵”建档，7个月，8.1kg，已出2颗牙 | 创建档案      | `create_or_update_baby_profile` |
| 切换宝宝 | 切到二宝                      | 切换上下文     | `switch_active_baby`            |
| 周计划  | 给朵朵安排下周辅食                 | 生成周计划     | `generate_weekly_meal_plan`     |
| 排敏   | 这周帮朵朵加蛋黄和芝麻               | 生成排敏计划    | `generate_allergen_intro_plan`  |
| 记录反馈 | 朵朵今天吃牛油果后嘴周发红，但很爱吃        | 记录反应并更新状态 | `log_food_reaction`             |
| 动态调整 | 根据这三天反应重新排下周菜单            | 重排计划      | `adjust_plan_by_feedback`       |
| 采购   | 把下周清单整理出来                 | 聚合食材      | `generate_shopping_list`        |
| 联动   | 把朵朵下周辅食加到苹果日历             | 导出日历      | `export_calendar_ics`           |


### 13.3 推荐输出模板

1. 当前宝宝：昵称、月龄、风险级别
2. 计划摘要：本周新增食材、常规食材、餐次数
3. 逐日菜谱：菜名、食材、做法、质地、观察点
4. 风险提示：禁用食材、需观察项目、何时停止尝试
5. 联动产物：采购清单、日历事件、提醒事项 payload
6. 规则依据：命中的关键规则与排除原因

## 14. 关键业务流程

### 14.1 首次建档流程

1. 用户输入宝宝基础信息。
2. 系统创建独立 `baby_profile`，自动计算月龄。
3. 系统补充询问已耐受食材、禁用食材与过敏史。
4. 若用户启用苹果联动，则记录偏好的输出方式（ICS、Shortcuts、文本）。

### 14.2 周计划生成流程

1. 读取 `active_baby` 档案与近 14 天 `food_trial`。
2. 计算当前可用食材池，过滤禁用、风险阻断与年龄不适宜形态。
3. 若本周包含排敏目标，插入新食材并预留观察窗口。
4. 按多样性、做饭便利度和宝宝接受度编排 7 天菜单。
5. 输出计划摘要、逐日菜单、采购清单与联动 payload。
6. 同步写入 `rule_audit_log` 以便解释与回归测试。

### 14.3 反应记录与动态调整流程

1. 记录某次进食的食材、反应、严重程度、爱吃度、排便与胃肠表现。
2. 刷新 `food_status`。
3. 若命中风险阻断规则，则中止继续安排该食材及相关高敏尝试。
4. 触发 `adjust_plan_by_feedback` 重排接下来 3-7 天菜单。

### 14.4 苹果生态联动流程

1. 从 `meal_plan` 读取计划结果。
2. 转换为 ICS、提醒事项 payload 或 Markdown 周报。
3. 若桥接失败，则保留文本/JSON 备选输出。
4. 外部联动失败不得影响计划生成主流程。

## 16. 风险控制、测试与运维考虑

### 16.1 风险控制


| 风险         | 等级  | 触发条件          | 系统动作                                    |
| ---------- | --- | ------------- | --------------------------------------- |
| 多宝宝串档      | 高   | 未明确操作对象       | 输出前强制显示 `active_baby`，写操作必须绑定 `baby_id` |
| 危险食材形态     | 高   | 窒息风险或年龄不适宜    | 阻断输出并给出替代形态                             |
| 严重过敏迹象     | 高   | 呼吸、面唇肿胀、持续呕吐等 | 停止推荐并进入风险提示流程                           |
| 规则与 LLM 失配 | 中   | LLM 输出与规则表冲突  | 以规则引擎裁决并记录审计日志                          |
| 外部联动失败     | 中   | Apple 导出或桥接失败 | 保留文本/JSON 备选输出                          |
| 文档再次分叉     | 中   | 多份文档并行维护      | 仅维护本文件为主文档                              |


### 16.2 测试策略


| 测试类型    | 覆盖范围                             | 优先级 |
| ------- | -------------------------------- | --- |
| 规则单元测试  | 月龄、频次、质地、高敏食材、阻断规则               | P0  |
| 状态机测试   | 8 种状态的合法流转路径                     | P0  |
| 多宝宝隔离测试 | 连续切换宝宝并写入日志，验证不串档                | P0  |
| 工具集成测试  | 各 Tool 的端到端调用链路                  | P1  |
| 联动测试    | ICS、Shortcuts payload、JSON 结构正确性 | P1  |
| 回归测试    | 固定样本档案的计划稳定性与规则命中                | P2  |


### 16.3 运维与数据建议

- 优先本地 SQLite 单文件存储，便于备份和迁移。
- 为规则表和食材库维护版本号，计划结果保存 `source_version`。

- 后续若接入云同步，应先定义隐私和加密方案，再开放。

## 20. 结论

本项目应被定义为一个以规则引擎为核心、具备多宝宝隔离、个体化记忆与苹果生态联动能力的“科学辅食操作系统”。

首版实现建议优先聚焦四件事：

1. 规则化
2. 可记忆
3. 不串档
4. 可导出

在此基础上，再逐步扩展排敏深度能力、自动调整和更强的苹果生态桥接。
